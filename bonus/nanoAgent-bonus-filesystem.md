# 从零开始理解 Agent（番外篇）：为什么 Agent 需要一个文件系统？

> **「从零开始理解 Agent」系列番外** —— 回顾第一篇，我们给 Agent 的三个工具是 `read_file`、`write_file`、`execute_bash`。当时没有解释为什么是这三个，而不是别的。这篇番外回答这个问题：文件系统和 bash 对 Agent 意味着什么？为什么生产级 Agent 都要在此基础上加一层沙箱？

---

## 一、第一篇的三个工具不是随便选的

第一篇给 Agent 定义了三个工具：

```python
TOOLS = [
    {"name": "read_file",     ...},  # 读文件
    {"name": "write_file",    ...},  # 写文件
    {"name": "execute_bash",  ...},  # 执行命令
]
```

这三个工具看似普通，但它们组合起来实际上给了 Agent 一个完整的**操作系统接口**：

- `read_file` = 感知环境（看到文件内容）
- `write_file` = 改变环境（创建或修改文件）
- `execute_bash` = 做任何事（因为 bash 能调用系统上的一切）

这不是 nanoAgent 的简化选择——业界也是这么想的。Claude Code 设置 `CLAUDE_CODE_SIMPLE=true` 后，就只保留三个工具：Bash、读文件、改文件。一个完整的编码 Agent，三个工具就够了。nanoAgent 第一篇选的工具集，恰好就是这个最小集合。

特别是 `execute_bash`——它不是"一个工具"，它是"一个可以调用所有工具的工具"。通过 bash，Agent 可以：

```bash
ls / find / grep       # 搜索和导航
git clone / commit     # 版本管理
pip install / npm      # 安装依赖
python / node          # 运行代码
curl / wget            # 访问网络
docker / kubectl       # 操作容器和集群
```

一个 `execute_bash` 工具，等于把整台机器的能力都交给了 Agent。

---

## 二、Unix 哲学和 Agent 天然契合

为什么 bash 对 Agent 这么好用？因为 Unix 的设计哲学恰好和 Agent 的工作方式高度吻合。

**Unix 哲学的核心：** 每个程序做好一件事，程序之间通过文本流（stdin/stdout）组合。

比如"找到当前目录最大的 5 个 Python 文件"，在 Unix 里是一条管道：

```bash
find . -name "*.py" | xargs wc -l | sort -n | tail -5
```

四个小工具串成流水线：找文件 → 统计行数 → 排序 → 取最大的五个。写这条命令的人必须**提前知道**每一步的输入输出格式，一次性把整条链写好。

Agent 做同样的事，过程不一样：

```
轮次 1: LLM 思考 → "先看看有哪些 Python 文件"
        调用 execute_bash("find . -name '*.py'")
        拿到输出: src/main.py, src/utils.py, tests/test_main.py ...

轮次 2: LLM 看到文件列表 → "有 8 个文件，统计一下行数"
        调用 execute_bash("wc -l src/main.py src/utils.py ...")
        拿到输出: 150 src/main.py, 80 src/utils.py ...

轮次 3: LLM 看到行数 → "最大的 5 个是这些，写个总结返回给用户"
        不调用工具，直接返回结果
```

**相同点：** 都是用小工具的组合完成任务，每个工具只做一件事（find 只找文件、wc 只数行数），工具之间传递的都是文本。

**关键区别：** Unix 管道是人提前写死的一条链，执行时不看中间结果；Agent 是每一步看到上一步的结果后，再决定下一步做什么。管道是"盲跑"，Agent 是"边看边走"。

这个区别带来一个重要的好处：Agent 可以处理**事先无法预判的情况**。比如 `find` 返回了 0 个文件，Agent 会意识到"这个目录没有 Python 文件"，换一个目录找或者直接告诉用户；而写死的管道只会继续执行，最后给出一个空结果。

这也解释了为什么 Agent 几乎不需要专门的"代码分析工具"或"项目管理工具"——bash + 文件系统已经提供了一切原语。`grep` 就是搜索，`sed` 就是替换，`diff` 就是比较，`git log` 就是历史。Agent 只需要根据每一步的结果，决定下一步用哪个。

## 三、文件系统是 Agent 的"外部记忆"

第二篇讲了 Agent 的记忆——用 Markdown 文件保存对话历史，下次启动时读回来。但文件系统的作用远不止存记忆。

对 Agent 来说，文件系统至少扮演三个角色：

**1. 工作区（Workspace）。** Agent 写代码、生成报告、保存中间结果，都要写到文件。文件是 Agent 行动的"痕迹"——没有文件系统，Agent 的所有工作都只存在于 `messages` 里，关掉就没了。

**2. 上下文的延伸。** context window 有限，但磁盘无限。Agent 可以把暂时不需要的信息写入文件，需要时再读回来。这就是为什么编码 Agent 处理大项目时会大量使用 `cat`、`grep`、`head` 这些命令——它不是把整个项目塞进 context window，而是按需从文件系统中"查询"信息。

**3. 多 Agent 协作的共享总线。** 第四、五篇讲的 SubAgent 和 Teams，多个 Agent 之间怎么传递数据？最简单的方式就是文件：Agent A 把结果写入 `output.json`，Agent B 读取它。文件系统是天然的共享存储，不需要额外的消息队列或 RPC 框架。

## 四、问题来了：Agent 能访问整台机器

回顾第七篇安全篇：`execute_bash` 可以执行 `rm -rf /`。我们用命令黑名单、用户确认、输出截断做了三道防线。但这三道防线都是**应用层**的——Agent 的代码在拦截，而不是操作系统在拦截。

应用层防护的问题是：它只能拦截它知道的危险。黑名单能挡住 `rm -rf /`，但挡不住：

```bash
# 用 Python 删文件，绕过 bash 黑名单
python3 -c "import shutil; shutil.rmtree('/')"

# 用 find + exec 删文件
find / -delete

# 用 dd 覆写磁盘
dd if=/dev/zero of=/dev/sda bs=1M
```

只要 Agent 有 bash 执行权限，理论上就能用无穷种方式绕过应用层的黑名单。这不是黑名单写得不好的问题，而是**应用层防护和 OS 层防护的根本区别**。

---

## 五、沙箱：用操作系统限制 Agent

生产级 Agent 的做法是：不在应用层"猜"哪些命令危险，而是让操作系统直接限制 Agent 能访问什么。

沙箱使用 OS 层面的原语来实现文件系统和网络的双重隔离。具体来说：

**文件系统隔离：** Agent 只能读写当前工作目录及其子目录。想访问 `~/.ssh/`、`/etc/passwd`、`~/.bashrc`？操作系统直接拒绝，不是 Agent 代码拒绝。

**网络隔离：** Agent 的所有网络请求必须通过一个代理服务器，代理服务器在沙箱外面运行，控制 Agent 能访问哪些域名。想把 SSH 密钥发到攻击者的服务器？代理直接丢弃请求。

这样，即使一次成功的 prompt 注入攻击也会被完全隔离，无法影响用户的整体安全。

用第七篇的 Hook 视角来理解：应用层的命令黑名单是一个 `before_tool_hook`，在 Agent 调用 bash 之前检查命令。而沙箱是在 Agent 调用 bash 之后、命令实际执行时，由操作系统进行检查。两层叠加就是"纵深防御"——权限系统是第一道门（"这个工具应该运行吗？"），沙箱是第二道门（"如果运行了，它能碰什么？"）。

## 六、沙箱带来的设计变化

有了沙箱之后，Agent 的安全模型从"拦截危险命令"变成了"定义安全边界"：

**没有沙箱时（第七篇的做法）：**
- 默认允许一切，然后逐个拦截危险操作
- 需要维护黑名单，永远有漏网之鱼
- 每条命令都要用户确认，确认多了就疲劳

**有了沙箱之后：**
- 默认禁止一切，然后逐个开放需要的权限
- 只需定义"Agent 能碰什么"，不需要猜"Agent 会做什么坏事"
- 沙箱在内部测试中将权限确认提示减少了 84%

这个变化的本质是：**从黑名单思维转向白名单思维。** 黑名单永远有遗漏（你能列出所有删文件的方式吗？），白名单天然安全（只有白名单里的路径可以写）。

---

## 七、回到 nanoAgent

nanoAgent 没有沙箱，这是有意为之——它是教学项目，不是生产工具。但理解了文件系统和沙箱的关系之后，你会看到 nanoAgent 的三个工具 + 第七篇的安全防线，和生产级 Agent 的沙箱架构之间，只差一步：

```
nanoAgent:     bash 工具 → 应用层黑名单 → 直接执行
生产级 Agent:  bash 工具 → 应用层 Hook → OS 沙箱 → 执行
```

多的那一层，就是操作系统的强制隔离。

文件系统给了 Agent 作用于真实世界的能力，Unix 哲学让这个能力可以无限组合，沙箱确保这个能力不会失控。三者合在一起，就是 Agent 和物理世界交互的完整架构。
